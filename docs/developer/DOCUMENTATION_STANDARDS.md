# 文档编写与维护规范

## 1. 命名规则
- **统一使用明确语义**：`<DOMAIN>_<TYPE>.md`，例如 `REFACTORING_GUIDE.md`（重构领域 + 指南类型）。
- **发布归档**：`docs/releases/RC_RELEASE_NOTES_<version>.md`，例如 `RC_RELEASE_NOTES_2.0.0-rc.md`。
- **状态报告**：`DOCUMENTATION_STATUS_REPORT.md`（状态对齐报告，单一入口）。

## 2. 元数据（文首）
每份文档头部必须包含以下元数据：
```markdown
# 标题
**版本**: <Version>
**日期**: <YYYY-MM-DD>
**状态**: <Status>（草案/RC/已发布/维护）
**责任人**: <Owner>
```

## 3. 格式标准
- **标题层级清晰**：使用 `#`、`##` 等。
- **要点短句化**：避免长段落，使用 `-` 列表。
- **示例使用 fenced code**：指定语言，如 `bash` 或 `python`。
- **文件引用**：使用 Markdown 链接，目标应使用相对路径。

## 4. 变更流程
- **功能变更**：提交 PR 时必须同步更新相关文档及 `DOCUMENTATION_STATUS_REPORT.md`。
- **RC 冻结**：统一文档标签为 RC，输出发布说明并归档到 `releases/`。
- **正式发布**：将 RC 发布说明转换为 Release Notes，并更新所有文档状态为“已发布”。
- **归档隔离**：活跃文档不得链接或引用已归档文档；归档内容只作为历史记录保留，不从用户入口、开发者入口、报告或发布说明回链。

## 5. 可读性与易用性
- **首屏摘要**：每份文档第一段应简述用途。
- **导航链接**：相关文档之间应有链接跳转。
- **快速上手**：示例代码应集中在用户向导文档，并保证可直接运行。
