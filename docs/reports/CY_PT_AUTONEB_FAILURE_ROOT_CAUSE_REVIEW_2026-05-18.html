Cy-Pt AutoNEB Failure Root-Cause Review

Date: 2026-05-18. Repository: /home/pku-jianghong/liuzhaoqing/work/deepmodeling/atst-tools. Branch inspected: develop. Case: examples/03_autoneb_Cy-Pt.

结论：当前分支在 SAI atst-dev + ASE 3.28.0 + ABACUS 3.10.1 + abacuslite 执行栈下，不能以 main-like 参数复现 main 分支已提交的 Cy-Pt AutoNEB 收敛结果。关键原因不是单一 AutoNEB 算法错误，而是旧执行栈与新执行栈的组合差异：ABACUS 输入兼容性变化、ABACUS 版本变化、legacy ase-abacus/abacus_autoneb wrapper 与 abacuslite wrapper 行为差异、MPI 启动模型差异，以及当前示例为稳定性 smoke run 主动采用了不同于 main 的 FIRE 稳定化参数。

main 分支历史结果成功并收敛
10 images，forward barrier 1.327886 eV，最高能量 image 5。
当前分支 as-is 示例能跑完但不复现
10 images，forward barrier 2.336211 eV，使用 downhill_check、maxstep: 0.05、maxsteps: 1。
当前分支 main-like 参数失败或失稳
严格参数因 xc pbe 被 ABACUS 3.10.1 拒绝；去掉 xc 后第一轮 FIRE 失稳。

3. main 分支成功运行的实际设置

main 分支案例位于 examples/Cy-Pt@graphene/autoneb。其 autoneb_run.py 直接构造 legacy AbacusAutoNEB，核心设置如下：

mpi = 16
omp = 4
neb_optimizer = FIRE
algorism = "improvedtangent"
climb = True
fmax = [0.20, 0.05]
n_simul = world.size
n_images = 10
parameters = {
    "calculation": "scf",
    "nspin": 2,
    "xc": "pbe",
    "ecutwfc": 100,
    "ks_solver": "genelpa",
    "smearing_method": "gaussian",
    "scf_nmax": 100,
    "init_wfc": "atomic",
    "init_chg": "atomic",
    "out_chg": -1,
    ...
}

其 Slurm 脚本加载的是旧环境：conda activate gpaw-intel、module load abacus/3.4.2-icx，并通过 mpirun -np $NSIMUL -machinefile slurm.hosts gpaw python autoneb_run.py 启动 Python 本身。因此 n_simul = world.size 是 MPI Python world，而不是普通串行 CLI 里的 YAML 数值。

4. 当前分支示例的实际设置

当前分支案例位于 examples/03_autoneb_Cy-Pt/config.yaml。它已转成统一 CLI + YAML 入口，通过 atst run config.yaml 调用 AutoNEBRunner。关键设置为：

calculation:
  type: autoneb
  n_simul: 4
  n_max: 10
  algorism: improvedtangent
  parallel: true
  optimizer: FIRE
  optimizer_kwargs:
    downhill_check: true
    maxstep: 0.05
  fmax: [1.00, 0.05]
  climb: true
  maxsteps: 1

calculator:
  name: abacus
  abacus:
    mpi: 4
    omp: 1
    parameters:
      ks_solver: cusolver
      scf_nmax: 300
      smearing_method: gauss
      pseudopotentials: ...
      basissets: ...

这些参数使当前示例成为“当前执行栈下可稳定完成的 smoke/stability 示例”，而不是历史 main 结果复现输入。特别是 downhill_check: true、maxstep: 0.05、maxsteps: 1 会显著改变 FIRE 每轮子优化的轨迹。

5. current branch 与 main branch 代码路径差异

层级 main branch current branch 对失败分析的影响
用户入口 手写 Python 脚本 autoneb_run.py。 统一 CLI atst run config.yaml。 当前分支更工程化，但运行模型、默认值、配置解释均不同。
AutoNEB 包装 source/abacus_autoneb.py 调用 forked my_autoneb.AutoNEB。 src/atst_tools/mep/autoneb.py 继承 ASE 3.28.0 AutoNEB，只替换内部 NEB 为 AbacusNEB。 当前实现更接近 ASE-native；main 历史结果来自 forked wrapper 的组合行为。
NEB 实现 forked AutoNEB 内部构造 ASE NEB。 构造 AbacusNEB，其 get_forces() 同步 ASE 3.28.0 并回移 ASE development 的 unconstrained real-force fix。 当前 NEB 算法路径不是主要失败源；它反而修复了 trajectory force 记录风险。
calculator 附着 每个 image 调用 set_calculator()，parallel 时目录为 AutoNEBrun-rank{world.rank}。 serial 非共享时目录为 autoneb_run/image_###；MPI parallel 时为 base-rank{rank}。 文件隔离策略不同；当前策略避免串行多 image 共享同一 ABACUS scratch。
结果冻结 forked my_autoneb 在每轮后调用 neb.distribute()。 当前实现恢复 store_E_and_F_in_spc / neb.distribute() 生命周期。 当前已修复此前可能破坏 AutoNEB 重启/加图状态的实现风险。
FIRE 参数 默认 FIRE，无 downhill_check、无显式 maxstep。 YAML 可透传 optimizer_kwargs；示例启用 downhill_check 与较小 maxstep。 这是当前分支能稳定跑完但不复现 main barrier 的直接参数原因之一。

6. ase-abacus 与 abacuslite interface 差异

main 分支使用的是 legacy ase-abacus fork，当前分支按项目重构方向使用环境内 abacuslite，失败分析中必须把两者视为不同 calculator interface，而不是同一接口的换名。

维度 legacy ase-abacus current abacuslite path 影响
加载方式 from ase.calculators.abacus import Abacus, AbacusProfile。 atst_tools.calculators.abacuslite_backend 优先导入环境 abacuslite，否则 fallback 到 vendored snapshot。 当前路径符合项目方向，但不能保证 legacy 输入/输出字节级一致。
资源文件键名 惯用 pp、basis、basis_dir。 规范键为 pseudopotentials、basissets、orbital_dir；factory 兼容转换 pp/basis/basis_dir。 键名可迁移，但写出的 INPUT/STRU 与复制/路径策略可能不同。
命令构造 legacy wrapper 中直接 AbacusProfile(command=f"mpirun -np {mpi} {abacus}")。 _build_abacus_command() 在 mpi > 1 时给非 MPI launcher 自动加 mpirun -np {mpi}。 当前更统一，但与旧 wrapper 的命令、stdout、版本解析路径不同。
版本解析 legacy get_abacus_version() 匹配 Version: 风格。 vendored abacuslite 已补丁兼容 ABACUS 3.10.1 banner-version 格式。 之前 SAI 验证暴露过 parser 兼容问题；已修复，但证明 wrapper 层确有行为差异。
ABACUS 参数接受性 main 历史环境接受或至少未因 xc pbe 中断。 ABACUS 3.10.1 明确拒绝 xc 参数。 严格 main 输入在当前执行栈无法启动，直接排除“同参数复现”。

7. ASE 版本、NEB 与 AutoNEB 算法差异

当前 atst-dev 环境中已实测 ase.__version__ == 3.28.0，NEB 与 AutoNEB 分别来自 .../site-packages/ase/mep/neb.py 与 .../site-packages/ase/mep/autoneb.py。ASE 3.28.0 相比旧版本的重要变化包括：NEB/DyNEB 默认方法从 legacy aseneb 方向迁移到 improvedtangent，并修复 improved tangent 中零范数 tangent guard。当前 YAML 和 main 脚本都显式使用 improvedtangent，所以默认方法差异不是本案例主因。

NEB 与 AutoNEB 的角色差别如下：

算法层 职责 本案例相关点
NEB 在固定 image 数量上做投影力优化，计算 tangent、spring force、climbing image force。 AbacusNEB 当前对齐 ASE 3.28.0，并补回 unconstrained real forces；该层没有显示出导致 main-like 失稳的独立 bug。
AutoNEB 先优化已有 images，再根据空间/能量 gap 动态加图，达到 n_max 后做 CI-NEB/refinement。 如果第一轮 NEB 子优化在当前执行栈下已经发散，AutoNEB 根本无法进入正常加图阶段。
FIRE optimizer 根据力和速度更新结构；downhill_check 和 maxstep 控制上坡/大步失稳风险。 当前 as-is 示例依靠稳定化 FIRE 参数完成；main-like 参数去掉这些保护后在当前执行栈失稳。

因此，本案例的失败更准确地说是“main-like AutoNEB/FIRE 参数在当前 ABACUS+wrapper 数值面上失稳”，不是“当前 ASE AutoNEB 算法实现与 main fork 有一个单点语义错误”。

9.2 严格 main 参数在 ABACUS 3.10.1 下直接失败

temp_sai_main_repro_validation/main_equiv_probe 保留 xc: pbe、ks_solver: genelpa、fmax: [0.20, 0.05]、无 downhill_check。job 423207 6 秒失败，ABACUS 报错：

THE PARAMETER NAME 'xc' IS NOT USED!
Bad parameter, please check the input parameters in file INPUT

这条证据很关键：在当前生产 ABACUS 环境中，“完全相同输入参数”的复现实验无法到达 AutoNEB 数值阶段。

9.3 去掉非法 xc 后第一步已不同

temp_sai_main_repro_validation/main_equiv_probe_no_xc 只删除 ABACUS 3.10.1 不接受的 xc，其余尽量保持 main-like。job 423218 捕获第一轮 FIRE 后取消。第一条真实优化记录已经不同：

10. 关键原因分层研判

  1. 硬阻断：main 的 xc pbe 在 ABACUS 3.10.1 中非法，严格同输入复现不可运行。
  2. 数值面已变：即使只删除 xc，第一轮真实 force evaluation 已与 main 不同，说明 ABACUS 版本、wrapper writer、默认解释或 SCF 路径已经改变。
  3. 优化稳定性不同：当前 as-is 能完成依赖 downhill_check 和较小 maxstep；main-like FIRE 设置在当前数值面上失稳。
  4. 运行模型不同：main 通过 MPI 启动 Python，world.size 直接决定 n_simul；当前普通 atst run 未 MPI 启动时会降级为串行 image 执行。
  5. wrapper 行为不同：legacy ase-abacus 与 current abacuslite 在参数规范化、输入文件生成、版本解析、目录策略、命令构造上均不同。
  6. 当前代码并非主要 bug 源：AbacusAutoNEB 当前已恢复 ASE schedule/result-freezing 语义，并修复串行 calculator 目录隔离；该层更像是把执行栈差异暴露出来，而不是制造 main-like 失稳的唯一原因。

11. 对项目下一步开发的建议

优先级建议理由
P0 不要在当前文档中宣称 examples/03_autoneb_Cy-Pt 可数值复现 main 历史结果。 严格输入不能运行；兼容输入已失稳；as-is 结果 barrier 差异约 1.0 eV。
P0 把当前示例定位为“ABACUS 3.10.1 + abacuslite 下的稳定 smoke/功能示例”，并在示例 README 或注释中解释其参数不同于 main。 避免用户把稳定化参数输出误读为历史 published/committed barrier 复现。
P1 保留 downhill_check、maxstep、maxsteps 等 YAML optimizer knobs，并继续测试其真实生效。 SAI diagnostic 已证明 downhill_check 会触发 FIRE reset，是当前执行栈稳定运行的关键工具。
P1 增加一个专门的 “legacy reproduction is not guaranteed” 文档段，列出 ABACUS 版本、wrapper、MPI 启动模型和参数兼容性条件。 这类科学计算 workflow 的可复现性依赖完整执行栈，而不只是 YAML 参数。
P1 如果必须做历史结果复现，应在隔离环境中重建 main stack：legacy ase-abacus、ABACUS 3.4.2、旧 conda env、MPI Python launch、原始 PP/ORB 路径和输入语义。 当前 SAI production stack 已经不是同一实验条件。
P2 继续压缩 AbacusNEB fork 面，待上游 ASE release 包含 real-force fix 后删除本地 get_forces() override。 减少未来 ASE 内部行为漂移的维护风险。

12. 最终判断

main branch 能成功运行、收敛并给出合理结果，是 legacy 执行栈整体行为的结果；当前 branch 失败，是因为用 main-like 参数迁移到新的 ASE-native + abacuslite + ABACUS 3.10.1 执行栈后，严格输入首先被 ABACUS 拒绝，最小兼容输入又在第一轮 FIRE 子优化中进入不可恢复失稳。

当前分支并非不能运行该案例：as-is 示例已在 SAI job 422164 中完成。但它完成的原因正是使用了不同于 main 的稳定化设置。因此，对项目下一步开发最有价值的结论是：把当前示例作为现代执行栈的稳定示例维护，而把 main 历史结果作为 legacy baseline 归档；除非完整重建 legacy stack，否则不要把两者的 barrier 数值设为相同验收目标。
