MACE-Reaction-Kit 到 ATST-Tools 的过渡态能力迁移研判

Date: 2026-05-27. Repository: /home/pku-jianghong/liuzhaoqing/work/deepmodeling/atst-tools. Scope: ATST-Tools current tree and temp_repos/mace-reaction-kit.

总论：不建议把 mace-reaction-kit 整体移植到 ATST-Tools。ATST-Tools 已覆盖 NEB、AutoNEB、Dimer、Sella、CCQN、D2S、Vibration 和 Sella IRC，真正值得吸收的是 MACE-Reaction-Kit 在“CCQN 自动化、TS 后验证流水线、结果产物治理”上的工程能力。迁移时应保持 ATST-Tools 的 CLI + YAML、ASE-native、ABACUS/DeePMD 后端定位。MACE model 可以作为可选 ASE calculator backend 评估，但不应引入 MACE 平台 runner 或默认 Torch/GPU 依赖。

优先吸收CCQN 自动化
自动枚举 reactive bond modes、product alignment、mode manifest、诊断日志。
谨慎吸收TS 后处理流水线
TS search 后接 vibration、IRC 或 descent path，需保持科学语义清晰。
谨慎评估MACE 后端
可考虑可选 calculator backend；不迁移 Adam/form runner 或 GPU CCQN 整体实现。

3. ASE 3.28.0 中 CI-NEB 的启动机制与 two-stage 的意义

本节专门回答三个容易混淆的问题：ATST-Tools 所称“支持 ASE NEB”是否包含 CI-NEB，ASE 是否原生内置“先跑几步普通 NEB 再开 CI-NEB”的调度，以及 MACE-Reaction-Kit 为什么还要额外写 two-stage NEB。结论是：ASE 3.28.0 的 NEB(climb=True) 已经原生支持 CI-NEB；但 ASE 原生 NEB 只提供 climb 开关，不自动执行 two-stage 调度。two-stage 的价值不是补齐 ASE 缺失算法，而是把“何时开启 climbing image”作为 workflow 策略管理起来。

3.1 ASE 3.28.0 的 CI-NEB 机制

在 atst-dev 环境中确认 ASE 版本为 3.28.0，源码文件为 /home/pku-jianghong/liuzhaoqing/.conda/envs/atst-dev/lib/python3.10/site-packages/ase/mep/neb.py。关键机制如下：

  • NEB.__init__ 接收 climb=False 参数，docstring 明确其含义为 “Use a climbing image”。
  • BaseNEB.__init__ 将该参数保存为 self.climb。
  • 每次 get_forces() 时，ASE 根据当前所有内点能量重新计算最高能量 image：self.imax = state.imax；NEBState.imax 的实现是对 energies[1:-1] 取最大值索引。
  • 当循环到 i == self.imax and self.climb 时，ASE 不再对该 image 加普通 NEB spring force，而是使用 full PES-derived force，并反转沿 tangent 的力分量：非 legacy 方法下执行 imgforce -= 2 * tangential_force * tangent。

因此，ATST-Tools 当前 calculation.neb.climb: true 的语义就是使用 ASE 原生 CI-NEB。ATST 的 AbacusNEB 继承 ASE NEB，并在自身 get_forces() 中保留了同样的 climbing-image 分支；这说明 ATST 对 ASE NEB 的支持包含 CI-NEB，而不是只支持普通 NEB。

3.2 “先普通 NEB 3-5 步再 CI-NEB”在 ASE 与 MACE 中的实现状态

从 ASE 3.28.0 源码看，原生 NEB 没有 two_stage、stage1_steps 或“先运行 3-5 步普通 NEB”的自动调度参数。ASE 的能力边界是：用户可以构造 NEB(images, climb=False)，用优化器先 run(..., steps=3) 或 run(..., steps=5)，随后手动设置 neb.climb = True 并继续优化。也就是说，ASE 支持这种做法所需的底层可变状态，但不把它封装成 workflow。

MACE-Reaction-Kit 则已经把这一实践封装成 workflow 级 two-stage。其 NEB workflow 先构造 NEB(images, ..., climb=(False if two_stage else climb), method="improvedtangent")；当 two_stage 启用时，第一阶段用普通 NEB 跑到 neb_stage1_fmax 或达到步数上限，然后执行 neb.climb = climb，再进入第二阶段 CI-NEB 精化。需要注意的是，MACE 的实现不是固定“3-5 步”，而是以较松 stage1_fmax 和全局 steps 为控制量；其 config/input.json 暴露了 neb_two_stage 和 neb_stage1_fmax，平台默认值为 false，workflow 代码在字段缺失时的 fallback 为 true。

3.3 stage1_fmax 是否优于固定 3-5 步

stage1_fmax 比单纯固定 3-5 步更接近 NEB 的物理目标。第一阶段普通 NEB 的目的不是消耗固定步数，而是让 band 先形成较合理的 MEP 形状和能量排序；用较松的力阈值，例如 0.1 或 0.2 eV/Ang，比“无论情况如何只跑 3-5 步”更能表达这个目标。

但是，MACE 的控制方式不应原样照搬到 ATST-Tools。MACE 第一阶段调用 dyn1.run(fmax=stage1_fmax, steps=self.steps)，第二阶段再调用 dyn2.run(fmax=fmax, steps=self.steps)。对 MACE/ML 势这通常可接受；对 ATST-Tools 的 ABACUS 后端，第一阶段和第二阶段都允许消耗完整全局步数会放大计算成本，也会让 max_steps 的用户语义变得不清晰。

因此，ATST-Tools 更合适的方案是采用 stage1_fmax + stage1_steps 双控制：stage1_steps 提供 3-5 步普通 NEB 预热的低成本默认行为，stage1_fmax 允许用户在愿意投入更多第一阶段步数时按较松力阈值停止。这个设计同时覆盖经验做法和收敛准则，也更符合 ABACUS 计算成本模型。

calculation:
  type: neb
  climb: true
  two_stage: true
  stage1_steps: 5
  stage1_fmax: 0.1
  fmax: 0.05
  max_steps: 100

推荐执行语义为：构造 NEB 时第一阶段强制 climb=False，运行 opt1.run(fmax=stage1_fmax, steps=stage1_steps)；随后设置 neb.climb = climb，第二阶段运行 opt2.run(fmax=fmax, steps=max_steps)。这样 max_steps 仍清晰表示最终 CI-NEB 阶段的步数上限，第一阶段成本由 stage1_steps 单独控制。

3.4 MACE-Reaction-Kit 为什么写 two-stage NEB

这个设计的理由在于 CI-NEB 的 climbing image 是“当前最高能量内点”。在初始插值路径还很粗、IDPP 尚未充分改善、或某个内点存在异常高能构型时，过早启用 climb 会立刻把这个最高能量 image 作为 climbing image，移除其 spring force 并反转切向力。这样可能使路径还未成形就开始追 saddle，增加最高能量 image 选错、路径扭曲或优化不稳定的概率。

two-stage 的工程含义是：第一阶段让普通 NEB 先建立较合理的 MEP 带形和能量排序；第二阶段再开启 CI-NEB，把已经较可信的最高能量 image 推向一阶鞍点。这是稳定性策略，不是 ASE CI-NEB 功能缺失的替代实现。

3.5 对 ATST-Tools 的迁移结论

问题 结论 对报告路线的影响
ATST 的 ASE NEB 是否支持 CI-NEB？ 支持。climb: true 直接映射到 ASE 3.28.0 的 CI-NEB 机制。 报告中不应把 CI-NEB 作为待迁移算法，只应把 two-stage 作为可选稳定性策略。
MACE 的 two-stage 是否是新算法？ 不是。它是普通 NEB 预优化后再打开 ASE CI-NEB 的调度策略；MACE 已实现，但不是固定 3-5 步。 建议在 ATST 中新增 neb.two_stage、neb.stage1_steps 和 neb.stage1_fmax，而不是新增新的 NEB backend。
MACE 的 stage1_fmax 控制是否更好？ 方向更好。它比固定步数更接近“普通 NEB 先达到较松路径质量”的目标。 ATST 应吸收 stage1_fmax，但必须同时引入 stage1_steps，避免 ABACUS 第一阶段成本失控。
two-stage 是否应默认开启？ 建议实现，但不建议立即改变 ATST 现有默认。ABACUS 场景计算成本高，且默认行为变化会影响既有 examples。 先作为显式 opt-in 选项加入；默认 stage1_steps: 5、stage1_fmax: 0.1 可作为模板推荐，而不是强行替代当前单阶段 CI-NEB。

4. DP 与 MACE 模型后端支持研判

ATST-Tools 当前已经把 DeePMD-kit / DP 作为正式 ML potential calculator backend 接入，而不是临时脚本级支持。相比之下，MACE-Reaction-Kit 的设计从一开始就围绕 MACE model 展开。是否在 ATST-Tools 中支持 MACE model，应按“新增一个可选 ASE calculator backend”来评估，而不是把 MACE-Reaction-Kit 的平台 runner 整体迁入。

4.2 MACE-Reaction-Kit 的 MACE 后端能力

MACE-Reaction-Kit 的模型后端集中在 Workflow.create_mace_calc()：根据 model_selection 或 custom_pes_model 选择模型，处理 device、dtype、multi-head、D3 修正等参数，然后构造 mace.calculators.MACECalculator。其配置表单还暴露了内置模型选项，例如 mace-omat-0-small.model 和 mace-mh-1.model。

这说明 MACE-Reaction-Kit 中的 MACE 支持很完整，但它和平台运行环境、内置模型路径、可视化输出、batch inference 以及 runner 表单高度耦合。ATST-Tools 若支持 MACE model，应抽取为轻量、后端无关的 calculator backend，而不是迁移这些平台绑定。

5. 任务运行信息收集与总结能力对比

除过渡态算法本身外，任务运行信息的收集、监控和总结也是 ATST-Tools 与 MACE-Reaction-Kit 的重要差异。两者都有类似能力，但形态不同：ATST-Tools 面向命令行和 HPC 离线运行，强调只读 summary CLI；MACE-Reaction-Kit 面向平台 runner，强调运行结束时返回标准化结果对象和文件路径。

4.1 ATST-Tools 当前 summary/collect 能力

ATST-Tools 的轻量命令不创建 calculator，不运行 ABACUS/DP，适合 Slurm 任务运行中轮询和完成后归档。核心实现位于 src/atst_tools/utils/summary.py 与 src/atst_tools/scripts/cli.py，并在用户文档中明确为 read-only summary 命令。

4.2 MACE-Reaction-Kit 的类似能力

MACE-Reaction-Kit 也有任务信息收集与总结能力，但它不是独立的离线 summary CLI，而是嵌入平台 runner 的结果返回机制。每个 workflow 返回 WorkflowResult，其中包含 summary、generated_files、plot_files、gifs 和 visualize_files。主 runner 随后生成 report.md，并按任务类型从 generated_files 中选择标准化输出，例如 TS 结构、trajectory、stream extxyz、IRC 两端、frequency report、mode table 和 CCQN mode manifest。

这个机制的优势是运行结束后平台可以直接展示状态和文件列表，且不同任务路线，例如 neb、neb_irc、sella_irc、ccqn_freq、ccqn_sella_irc，都有各自的 output map。其局限是它依赖 runner 调用链和约定文件名；对已经在 HPC 上独立运行、需要后续轮询或手动归档的任务，不如 ATST 的 read-only CLI 灵活。

4.3 迁移判断

问题 确认结果 建议
ATST summary CLI 是否可收集各任务信息？ 基本可以。它覆盖 NEB/AutoNEB、Relax、Dimer、Sella、CCQN、Vibration、D2S、ABACUS run directory 和 trajectory 工具。 保持 read-only 设计，继续作为 Slurm 监控和离线后处理主入口。
MACE 是否有类似功能？ 有，但形态不同。MACE 通过 WorkflowResult、report.md 和 runner output map 完成运行结束时的结果注册与展示。 不要把 MACE runner 搬进 ATST；吸收其“标准化产物注册表”的思想。
ATST 还缺什么？ 缺统一的跨 workflow artifact manifest。当前 summary CLI 能读结果，但工作流运行时不总是生成机器可读的产物索引。 新增后端无关 atst_artifacts.json 或 workflow_manifest.json，记录结构、轨迹、summary、validation、IRC、thermo、log 文件路径。

6. 值得实现的内容分级

P0：近期最高优先级

内容 为什么值得做 ATST 中的建议落点 验收标准
CCQN reactive bond modes 自动枚举 当前 ATST ic 模式要求用户手写 reactive_bonds，对表面催化反应门槛高。 新增 atst_tools.utils.reactive_modes，在 standalone CCQN 与 D2S-CCQN 中可选启用。 给定 molecule/support 划分、cutoff 和 max_modes 后，稳定生成去重排序的 1-bond/2-bond mode 列表。
CCQN product alignment interp 模式依赖 product-like structure，原子顺序不一致会导致错误 e-vector。 扩展 CCQN config：align_product_indices: true，复用现有 atom alignment/MIC 工具。 相同元素计数但索引打乱的 product 能被重排；元素计数不一致时明确报错。
CCQN mode manifest 多 mode 或自动枚举后必须知道每个候选尝试了什么、结果如何。 输出 ccqn_mode_manifest.json，记录 mode bonds、标签、energy、fmax、是否收敛、输出路径。 单 mode 和多 mode 均有 manifest；失败 mode 不吞错，记录异常摘要。
统一 workflow artifact manifest ATST 已有丰富 summary CLI，但缺少运行时统一产物索引；MACE 的 output map 对结果展示和平台集成很有启发。 每个 workflow 可选写出 atst_artifacts.json，summary CLI 优先读取并回退到现有路径推断。 NEB、D2S、CCQN、Vibration、IRC 至少记录主 trajectory、最终结构、summary JSON、validation JSON、log 和阶段状态。
CCQN diagnostics CCQN 失败往往需要判断 uphill/PRFO 切换、负本征值、trust radius 行为。 增强 CCQNOptimizer 日志和可选 JSON trace。 每步记录 mode、min eigenvalue、trust radius、rho、e-vector overlap、max force。
统一 TS validation summary Dimer/Sella/CCQN/NEB 产出的 TS 都需要一致判断“是否像一阶鞍点”。 在 vibration 基础上新增 ts_validation.json 生成器。 输出虚频数量、主虚频、最大力、能量、判断状态和主要文件路径。

P1：有价值但需控制语义

内容 收益 风险控制
Fake-IRC / descent backend Sella IRC 不稳定或昂贵时，可沿虚频方向双向下降，快速得到路径端点猜测。 命名为 descent 或 fake，文档明确“近似路径，不等价于严格 IRC”。
TS search → vibration → IRC 复合后处理 把用户常用的“找 TS、验虚频、跑 IRC”连成可复现工作流。 通过 post 段显式启用，不改变现有单任务默认行为。
NEB two-stage CI ASE 已原生支持 CI-NEB，但没有内置 two-stage workflow；MACE 已封装普通 NEB → CI-NEB，并用 stage1_fmax 表达第一阶段路径质量。 新增 neb.two_stage、neb.stage1_steps、neb.stage1_fmax；默认保持当前单阶段行为。不要照搬 MACE 的全局 steps 控制，ATST 第一阶段应单独限步。
endpoint relax 显式选项 对 ML 势和粗初猜更友好。 必须显式启用，并写清楚会改变端点结构；ABACUS 高成本场景要给出默认关闭策略。

P2：中期增强

  • CCQN batch/job-array：MACE 的 batch pairs 对高通量筛选有价值，但 ATST 的 ABACUS 场景不能简单串行照搬，应与 Slurm job array 或分任务 manifest 结合。
  • IS/TS/FS reaction thermo：在 TS validation 稳定后，基于 vibration 结果生成反应能垒、ZPE 修正和温度网格自由能表。
  • HTML/CSV summary：统一 examples 和生产任务的结果摘要，便于人工审查和文档归档。
  • MACE calculator backend：作为可选 ML potential 后端支持 calculator.name: mace，复用 ATST calculator factory 和 workflow，不迁移平台 runner。

P3：不建议近期实现

  • GPU/Torch CCQN 整体移植：ATST 的主要成本通常在 calculator，尤其是 ABACUS 调用；先引入 GPU optimizer 栈会增加依赖和维护面。
  • MACE 平台 runner：task_family、form config、output map 适合 Adam 平台，不符合 ATST 的 CLI + YAML 主接口。
  • MACE-Reaction-Kit 的平台绑定模型选择逻辑：内置模型列表、平台路径和 outputShow 不应进入 ATST；只应保留 calculator backend 层。
  • 替换现有 IDPP/alignment：ATST 已有 Fast_IDPPSolver 和 atom alignment，可补质量检查，不应重写路线。

calculation:
  type: ccqn
  init_structure: ts_guess.traj
  e_vector_method: ic
  auto_reactive_bonds:
    enabled: true
    molecule_indices: "1-12"

    vibration:
      enabled: true

      enabled: true
      backend: sella

对于近似路径，可显式使用 backend: descent，并在输出中写明它不是严格 IRC：

calculation:
  type: irc
  init_structure: ts_opt.traj
  backend: descent
  mode_vector: first_imaginary_mode.npy
  descent_delta: 0.1
  direction: both

10. 风险与边界

风险 说明 规避方式
科学语义混淆 Fake-IRC/descent path 容易被用户误认为严格 IRC。 接口命名、summary 和文档中明确标注 approximate descent path。
接口膨胀 MACE 的平台表单参数很多，直接搬入会破坏 ATST YAML 简洁性。 只暴露必要参数；高级参数放入嵌套段并保持默认关闭。
产物索引与 summary CLI 重复 新增 artifact manifest 后，若与现有 read-only summary 路径推断各自为政，会造成两套结果语义。 manifest 只记录产物位置和阶段状态；数值摘要仍由 summary 工具读取轨迹/cache 生成，必要时把 summary JSON 路径写入 manifest。
后端耦合 MACE-Reaction-Kit 的能力依赖 MACECalculator、平台模型路径和平台输出约定。 算法和 workflow 工具保持 calculator-neutral；MACE 只作为可选 ASE calculator backend 接入。
ML 后端依赖膨胀 DP 已依赖 deepmd-kit；MACE 会继续引入 Torch/MACE 栈。如果默认安装包含所有 ML 后端，会提高环境复杂度。 保持 DP/MACE 均为运行时可选依赖；MACE 使用 optional extra 和清晰 ImportError，不影响 ABACUS/DP 用户。
计算成本失控 auto mode enumeration 和 batch 容易产生大量 ABACUS 任务。 设置 max_modes、dry-run manifest、Slurm job-array 计划和失败恢复机制。

11. 最终建议

ATST-Tools 的近期演进应围绕“把已有过渡态算法变得更易用、更可验证、更可追踪”，而不是扩张算法名录。最值得从 MACE-Reaction-Kit 吸收的是 CCQN 自动 reactive mode、模式级 manifest、TS validation 和后处理 pipeline。GPU CCQN 和 MACE runner 不应进入近期路线；MACE model 支持可以作为独立的可选 calculator backend 任务评估。

建议下一步先拆出一份 Phase 1 实施计划：只做 CCQN 自动化增强，并同步修正文档中 D2S/CCQN 状态不一致的问题。这个阶段完成后，ATST-Tools 对单端 TS 搜索的用户体验会有实质提升，同时不会破坏当前 ABACUS/DP 工作流主线。
