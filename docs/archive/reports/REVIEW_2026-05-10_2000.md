## 1. Vibration Cache 管理机制审查

### 1.1 当前实现分析

从 `src/atst_tools/workflows/vibration.py` 的 `_prepare_cache` 方法分析：

```python
def _prepare_cache(self):
    vib_path = Path(self.name)
    if not vib_path.exists():
        return
    cache_files = list(vib_path.glob("cache*.json"))
    if self.restart:
        return
    for cache_file in cache_files:
        cache_file.unlink()
```

**实现结论**: 当前代码清理**所有 cache*.json 文件**，无论文件大小或内容状态。并非只清理非0字节文件。

### 1.2 Restart 机制设计

配置选项：`calculation.restart: bool`

- **默认值**: `False` → 清理旧cache
- **设置为 `True`**: 保留现有cache文件，实现续跑

**续跑最佳实践建议**：

1. **确认计算中断点**: 先检查已完成的位移方向和未完成的方向
2. **避免结构变动**: 确保 `init_structure` 与中断前一致
3. **保留 vib 文件夹**: 不要手动删除或重命名振动计算文件夹
4. **配置示例**:

```yaml
calculation:
  type: vibration
  init_structure: ts_opt.stru
  restart: true  # 启用续跑模式
  # ... 其他参数保持一致
```

### 1.3 改进建议（可选）

当前设计已足够解决 SAI 实算中暴露的问题。如需更精细的清理策略，可以考虑：
- 仅清理0字节的损坏cache文件
- 保留完整的cache文件但重命名备份

## 2. D2S Workflow 实现深度审查

### 2.1 Double-Ended 阶段实现

从 `src/atst_tools/workflows/d2s.py` 分析：

**已实现**:
- **端点优化**: 支持对初始和最终态进行几何优化
- **粗糙 DyNEB**: 使用 ASE 的 `DyNEB` 构建粗略路径
- **NEB 端点计算器**: 已修复！端点也会挂上计算器，避免读取能量时报错
- **IDPP 初始路径**: 使用 Fast_IDPPSolver 生成初始猜测

**当前限制**:
- **仅支持 DyNEB**: Double-Ended 阶段**不支持常规 NEB 或 AutoNEB**
- **设计原因**: D2S 设计目标是快速粗糙 NEB → 精细单端搜索，DyNEB 更适合这个场景

### 2.2 AutoNEB 支持情况

**AutoNEB 当前状态**: ✅ 完全独立支持，通过 `calculation.type: autoneb` 调用

**AutoNEB 实现分析** (`src/atst_tools/mep/autoneb.py`):
- 继承自 ASE `AutoNEB`
- 自定义 `_execute_one_neb` 方法使用 `AbacusNEB`
- 支持迭代历史备份
- 支持磁盘空间清理（删除临时计算目录）
- 支持并行执行

**D2S vs AutoNEB 关系**:
- **D2S**: 端点 → DyNEB → 单端搜索（设计目标）
- **AutoNEB**: 独立的自适应图像增加NEB方法（独立使用）

### 2.3 D2S 配置示例

```yaml
calculation:
  type: d2s
  method: dimer  # 或 sella
  init_file: init.stru
  final_file: final.stru
  
  neb:
    n_images: 10
    fmax: 0.20
    max_steps: 50
  
  dimer:
    fmax: 0.05
```

