概述

ATST-Tools 2.2.0 候选已提供稳定的 atst_tools.api、配置型 run_workflow()、嵌入式 run_ccqn()、结构化 WorkflowResult、CLI 兼容适配和外部启动 MPI 的同步失败处理。ABACUS Agent 目前仍以 vendored ATST 2.1.3 和 CLI subprocess 执行 transition workflow，尚未消费 API。

本设计在未发布的 2.2.0 中增加一个通用、无 ATP/SAI 依赖的 API runner 与 JSON 结果协议。ABACUS Agent 保留 YAML 生成、ATP Tool 输入输出、外层 MPI、资源申请和 result-evidence；它只将执行目标从 ATST CLI 替换为 runner。

目标与非目标

目标

  • 让任意外部宿主进程以 wheel 可安装的方式调用配置型 ATST API。
  • 为串行和外部启动的 image-parallel workflow 提供一致的 workdir、退出码、JSON 结果和错误契约。
  • 让 ABACUS Agent 不使用 ATST 私有模块或 ATST CLI 即能运行现有 transition workflow。
  • 保持 YAML、CLI、文件名、六个根导入、backend 选择和 MPI 责任边界。

非目标

  • 不启动 Slurm、mpirun、srun 或嵌套 ABACUS MPI。
  • 不把 adam-community、ATP tool_data、SAI 或 result-evidence 引入 ATST。
  • 不新增 ABACUS calculator factory；配置型 CCQN 使用 run_workflow，run_ccqn 仍要求调用方提供 Atoms 与 calculator。
  • 不改变既有 CLI 行为。

决策与备选

决策：ATST 拥有 runner 和 API result/error document；ABACUS Agent 拥有 ATP adapter、外层 MPI、SIF 与平台证据。

Agent 自建 runner：拒绝，会复制 CWD、错误链、root-only 写入和 MPI 退出规则。

继续调用 CLI：拒绝，不能消费稳定 Python API，仍耦合终端输出。

ATST 启动 mpirun/Slurm：拒绝，违反 API 边界并与平台资源生命周期冲突。

架构与数据流

ABACUS ATP Tool
  → YAML + Agent workdir + resource allocation
  → serial: python -m atst_tools.api.runner ...
  → parallel: mpirun -np N python -m atst_tools.api.runner ...
  → runner: chdir(workdir) → run_workflow(config, RunOptions(...))
  → root: atst-api-result-v1.json + ATST artifact manifest
  → Agent: JSON/manifest → result_evidence.json + ATP outputs

runner 支持 --config、--workdir、--result-json、--dry-run、--restart、--check-input、--check-input-timeout 和 --abacus-executable。并行时每个 rank 使用同一 runner，只有 root 在 API 成功同步后原子写 JSON。manifest 仍是科学产物的持久权威；JSON 是宿主进程交接文件。

模型层提供 WorkflowResult.to_document(workdir) 和 ATSTAPIError.to_document()，但不增加根导入。runner 通过这些方法生成 JSON，避免 Agent 依赖 dataclass 内部布局或异常字符串。

ATST-Tools 2.2.0 实现边界

  • 新增 src/atst_tools/api/runner.py：仅依赖 atst_tools.api 的公开服务和公开模型；用 argparse 实现 runner 参数，独立进程内建立 workdir，调用 run_workflow()，并映射稳定退出码。
  • 扩展 src/atst_tools/api/models.py：为 WorkflowResult 和 ATSTAPIError 提供 JSON-safe document 方法。document 只能包含 schema、状态、workflow、is_root、workdir、manifest、artifacts、metadata、错误分类和因果摘要；不得序列化 ASE Atoms 或泄露环境变量。
  • runner 在 API 调用完成及其 MPI 失败同步之后，仅让 root 以“同目录临时文件 + replace”写 --result-json；非 root 不创建该文件。成功 document 引用 API 已生成或工作流已写入的 manifest，失败 document 绝不伪造完成 manifest。
  • 新增 runner 参考文档和可执行最小示例；更新 release notes、README/API reference 和版本治理账本。runner 不成为第七个根导入，也不改动 atst console script。

ABACUS Agent 1.2.8 基线的承接方式

  • 保留 abacus_transition_state.py 中已暴露的 abacus_worker_transition_neb、abacus_worker_transition_autoneb、abacus_worker_transition_sella、abacus_worker_transition_ccqn、abacus_worker_transition_input_prepare、abacus_worker_transition_run 和只读的 abacus_basic_transition_summary Tool；不新增必须由用户理解的 runner Tool。
  • 在 toolkits/transition_atst.py 将当前 CLI subprocess adapter 替换为 runner subprocess adapter：准备阶段继续写 YAML；执行阶段组装 runner command；读取 result JSON 与 manifest；返回既有 TransitionWorkflowResult；任何 stdout/stderr 仅用于诊断。
  • 保持每个具体 Tool 直接继承 Tool、紧贴 class 的 docstring、既有 calltype="python"、直接赋值的 DISPLAY_NAME/CONDA_ENV/SIF/NETWORK/CPU/GPU/MEM_PER_CPU/PARTITION，以及既有 @tool_io output ports。不得把 result JSON 作为破坏兼容性的必填 Tool 参数或新增 DAG port。
  • Agent adapter 将 result JSON 转为现有 result-evidence 和 registered files；result JSON 作为内部工作目录文件加入 display/track 路径，稳定跨 Tool 交接仍使用显式文件路径与项目 state，而不是把 DAG outputs 复制进 tool_data。
  • 串行执行启动 runner；image-parallel NEB/AutoNEB 保留现有外层 mpirun -np N、SAI-compatible launcher 与 rank/GPU 校验，只将子命令从 CLI 改为 runner。CCQN 第一阶段保持 YAML/run_workflow route；不在本任务中把调用方 calculator factory 下沉至 ATST。

错误与 MPI

runner 捕获 ATSTAPIError；root 写包含 error type、workflow、context、cause type/message 的失败 document 并返回非零状态。所有 rank 必须先完成 run_workflow 的失败同步，才可进入 root-only JSON 写入。JSON 以同目录临时文件加 replace 原子落盘。runner 不伪造成功 manifest；失败不会报告完成状态。

测试与验收

ATST 本地与 wheel 门禁

  • ATST unit：runner 参数、workdir 恢复、success/failure document、非 root 不写 JSON、CLI 未变化、document 不含 ASE 对象。
  • ATST clean-wheel：基础 runner dry-run；安装 [parallel] 的 mpiexec 两 rank success/failure；禁止 system-site-packages 与源码导入。
  • ATST real MPI：配置、calculator、NEB/AutoNEB engine/optimizer/writer 失败经实际 runner 进程触发且无 timeout；保留 root cause 和 peer 退出。

ABACUS Agent 本地 ATP 门禁

  • 以 1.2.8 作为行为基线，所有 Agent 本地命令均在 conda run -n abacus-env 下执行；先递归初始化记录的 Git 子模块。新增/更新 tests/test_atst_transition_tools.py 和 result-evidence tests，验证 CLI command 已替换为 runner command、JSON-to-evidence、串行、restart、dry-run、NEB/AutoNEB outer MPI 与失败显示。
  • 执行 conda run -n abacus-env pytest tests -q、conda run -n abacus-env make parse、conda run -n abacus-env make build 和 conda run -n abacus-env python3 scripts/check_build_package_contents.py。parser output 必须继续暴露所有 transition Tool、原参数和直接 class metadata；build zip 不得含 credentials、token 或临时文件。
  • 运行 conda run -n abacus-env python3 scripts/check_abacus_tool_integrity.py --root . --parser-command "conda run -n abacus-env make ADAM_CLI=adam-cli parse"；对修改的 Tool 再用 parser API 做单文件检查。DAG output 的返回 dict keys 必须与既有 @tool_io(outputs=...) 完全一致。
  • 更新至少一个现有 ATST demo 的 package-relative fixture 和用户可理解的 message；demos/demos.json 条目保留 name、description、onlineUrl、_taskid、message、files 六个字段，且不暴露 runner/result JSON 等内部术语。

SAI GPU/Slurm 验证

  1. 登录节点只用于同步源码、构建、编辑、提交和查询。基于 /opt/sbatch_examples/gpu_abacus.sbatch 创建独立验证脚本；使用 4V100 ABACUS LTS 3.10.1 基线，--nodes=1 --ntasks=4 --gpus-per-node=4，并 source /opt/sai_config/mps_mapping.d/${SLURM_JOB_PARTITION}.bash。
  2. 串行 API runner：运行现有 H2/Au relax YAML，验证 result JSON、artifact manifest、ABACUS version、final energy 和 sacct 的 COMPLETED|0:0。
  3. image-parallel：使用既有 Agent/ATST NEB 或 AutoNEB fixture，申请与 interior images 或 n_simul 相等的 GPU/rank；外层 mpirun 使用与 ABACUS LTS 同源 OpenMPI，runner 的每个 rank 同时调用 API。验证 root-only JSON、所有 rank 退出、manifest 和 Agent evidence。
  4. 保留现有 CLI H2/Au 任务作为回归对照；仅当 CLI 与 runner 两条路径均完成、产物定位一致且没有错误日志时才能通过。
  5. 通过 squeue 监控，结束后使用 sacct -j <jobid> --format=JobID,State,ExitCode、Slurm stdout/stderr、ABACUS output、runner JSON、manifest 和 result-evidence 形成证据。不要在登录节点运行计算；避免在共享 home 进行高频小文件压力测试。

AsterFire ATP CLI 平台验证

  1. 先做只读 readiness：command -v asterfire、asterfire version --json（必须为 0.3.5），再令任务专用变量 ASTERFIRE_DEV_HOME=/home/james/.local/share/asterfire/profiles/dev-home，执行 HOME="$ASTERFIRE_DEV_HOME" asterfire me --json 和 HOME="$ASTERFIRE_DEV_HOME" asterfire toolboxes list --json。该 HOME 仅在单条命令前注入，不修改 shell 的常驻 HOME；不在仓库、tmp_runs、文档或日志保存 token。
  2. 仅在本地 ATP gate、SAI/SIF gate 和 make build 成功后，且获得明确的上传授权时，执行 HOME="$ASTERFIRE_DEV_HOME" asterfire toolboxes upload <agent-1.2.9.zip> --json。上传后重新读取 toolbox list，确认 name/version/visible 与 config/configure.json 一致；不得用目录路径或未完成的 build 目录代替 zip。
  3. 经明确的文件上传与建任务授权后，执行 HOME="$ASTERFIRE_DEV_HOME" asterfire cloud upload <fixture> --id-only，保存本次返回的 file id，再以 HOME="$ASTERFIRE_DEV_HOME" asterfire tasks create --toolbox ABACUS --message <demo-message> --file <file-id> --json 创建任务。不要用旧 onlineUrl/_taskid 代替当前 fixture。轮询 tasks status 至终态，再使用 tasks get 检查 tool steps、最终回复、result JSON/manifest/result-evidence 文件交接。
  4. 生产平台操作必须改用经确认的 prod profile，并先确认该 profile 下 asterfire me --json 成功；dev 验证不等同生产发布。若 task 仍在运行、step 失败或只证明 CLI 连通，不能报告发布就绪。

发布与迁移

  1. 在未发布的 ATST 2.2.0 分支实现 runner、document、测试和文档，独立审查 API signature、CLI parity、wheel 和 MPI evidence 后合并。release note 合并前不得声称 branch 为 main。
  2. 构建、clean-install 并验证 2.2.0 wheel；创建准确 tag/release。Agent 不得依赖未发布 commit、可变本地 PYTHONPATH 或 ATST 私有模块。
  3. 从 ABACUS Agent 1.2.8 创建 1.2.9 候选：更新 deps/atst-tools、config/configure.json、Layer 2/SIF version checks、dependency audit、SIF status 与 build references；让 Layer 2 在 image-parallel 支持 [parallel] extra。
  4. 先在 Agent 串行 transition Tool 接入 runner，再接入 image-parallel Tool；每一步保留可验证的 CLI fallback，作为迁移期回滚路径而非双实现长期常态。
  5. 完成本地、SAI/SIF 和经授权的 AsterFire dev 平台验证后发布 Agent。生产 toolbox 上传、image 更新和正式 Agent task 是独立授权动作。

风险与开放项

风险：CWD 只能在 runner 独立进程中改变；MPI JSON 写入必须晚于 API 同步；SIF mpi4py 必须绑定 LTS 同源 OpenMPI。分别以进程隔离、root-only 原子写入和既有 Layer 2 校验缓解。

开放项：无阻塞架构决策。JSON 文件名、runner 模块名称和 Agent evidence 字段将在实施计划中按项目命名规范固定。
