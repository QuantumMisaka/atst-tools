Summary

ATST-Tools currently governs calculator-backed workflows through the atst CLI and YAML. Its package root exports only version helpers; workflow, calculator, and utility modules are implementation APIs with no external compatibility promise. This design adds a stable, versioned Python API under atst_tools.api.

The API has two complementary entry points. run_workflow() executes every existing YAML workflow with the same schema, defaults, calculator selection, filesystem behavior, and MPI semantics as atst run. run_ccqn() is a focused embedded ASE API for the ABACUS ASE-interface use case: a caller supplies Atoms and an already-created calculator, such as abacuslite, and ATST performs its distinctive CCQN transition-state search without YAML or calculator reconstruction.

Compatibility principle: the work is additive. No existing CLI command, YAML field, default, relative-path behavior, output file name, existing import path, MPI rule, or exit-code contract may change. The CLI becomes a thin adapter over shared internal services; it does not become a second workflow implementation.

Goals and non-goals

Goals

  • Provide a documented, tested, stable public namespace: atst_tools.api.
  • Make the existing YAML workflow surface callable from Python without invoking a subprocess.
  • Make atst run and atst config validate use the same normalization and execution services as the public API.
  • Provide a single-file, CCQN-specific embedded ASE API suitable for an optional ABACUS ASE-interface example.
  • Preserve image-level MPI semantics for NEB and AutoNEB, including root-rank result ownership.
  • Expose structured outcomes, artifacts, and errors rather than requiring callers to parse terminal output.
  • Publish user-facing API documentation, an executable API example, and clear CLI/API selection guidance as part of the same release.

Non-goals

  • Do not replace the CLI, YAML inputs, examples, or current workflow classes.
  • Do not make every internal utility, calculator factory, workflow class, or vendored abacuslite function public.
  • Do not create an atst Python package alias; atst remains the console command and atst_tools remains the import root.
  • Do not introduce MkDocs, Sphinx, pdoc, or another documentation-build system in the first API release.
  • Do not launch Slurm, mpirun, srun, or nested ABACUS MPI processes from the API.
  • Do not promise direct Atoms + calculator APIs for NEB, AutoNEB, D2S, MD, or DMF in the first release.
  • Do not change the experimental status or safety guards of DMF.

Decisions and alternatives

Selected public surface:

from atst_tools.api import (
    CCQNOptions,
    RunOptions,
    WorkflowResult,
    validate_config,
    run_workflow,
    run_ccqn,
)

Alternative rejected: expose existing workflow classes. Their constructors, calculator ownership, return values, and filesystem side effects are intentionally workflow-specific. Publishing them would freeze internals before they have a coherent compatibility contract.

Alternative rejected: retain CLI as the implementation and invoke it from Python. Subprocess execution cannot provide calculator injection, structured in-memory results, or reliable MPI error propagation, and would preserve two independently testable interfaces rather than one service layer.

Alternative rejected: create an atst import alias. It conflates the console-script name with the distribution import root and creates a second public namespace without a technical need.

Architecture and data flow

CLI arguments ──> CLI adapters ─┐
                                  ├─> internal application services ─> existing workflows / ASE / calculators
Python callers ─> atst_tools.api ─┘
                                             │
                                             └─> WorkflowResult + artifacts + typed exceptions

Configuration-driven execution

validate_config(config_source) accepts a YAML path or a mapping, validates through the existing schema, applies exactly the existing defaults, and returns a copied normalized mapping. run_workflow(config_source, options=RunOptions(...)) accepts the same sources. RunOptions contains only CLI-equivalent controls: dry_run, restart, ABACUS check-input settings, and an optional communicator.

Relative paths retain current semantics: they are interpreted from the calling process current working directory. The API must not silently change directory to the YAML parent. Callers needing a different location use absolute paths or establish their own working directory before calling, exactly as current examples do.

Embedded CCQN execution

run_ccqn(atoms, calculator, options=CCQNOptions(...)) copies the supplied atoms, attaches the supplied calculator to that copy, and delegates to shared CCQN execution logic. CCQNOptions covers only the CCQN parameters already represented by the schema: convergence, optimizer output paths, reactive bonds or automatic reactive-mode selection, product reference, and diagnostics/manifests. It does not accept an ATST calculator name or rebuild a calculator through CalculatorFactory.

Backend delegation invariants

  • Configuration-driven ABACUS workflows use the current backend resolver unchanged: import independently installed abacuslite first, then fall back to the ATST vendored snapshot only when that import is unavailable.
  • For run_ccqn(), the supplied ASE calculator is the sole backend owner. ATST attaches it to its private atoms copy but does not replace its profile, command, pseudopotentials, orbitals, working directory, MPI configuration, or I/O implementation.
  • The public API accepts an ASE calculator protocol; it neither republishes nor promises compatibility for individual abacuslite I/O functions or vendored modules.
  • Configuration-driven ABACUS results record backend_source: external or backend_source: vendored in stable metadata. Calculator-injection results record backend_source: provided.
  • The API remains a workflow, schema, artifact, and orchestration layer. It does not replace ABACUS, abacuslite, an ASE calculator, an MPI launcher, Slurm, or site-specific job launchers.

Workflow result

Every execution entry point returns WorkflowResult with workflow, status, is_root, artifact_manifest, artifacts, and JSON-compatible metadata. Single-structure workflows expose final_atoms. Path workflows expose final_images and ts_atoms; the latter is the highest-energy image when applicable. The artifact manifest remains the durable source of truth across processes and restarts.

To avoid falsely claiming immutability for mutable ASE objects, WorkflowResult is a frozen container but its returned Atoms are documented snapshots owned by the caller. API implementations return copies where practical; persisted files and JSON metadata remain the reproducible records.

User documentation

The existing documentation system is Markdown-first and already has maintained user entry points for installation, CLI, YAML, examples, and the ABACUSLite wrapper. It has no public Python API reference and no API-documentation generator. The first API release extends the existing governed Markdown path; it does not publish raw internal docstrings or introduce a second documentation toolchain.

The reference must explicitly say that imports from atst_tools.workflows, atst_tools.mep, atst_tools.calculators, and vendored modules are not the stable integration surface. The example and reference must use from atst_tools.api import ..., never from atst import ....

The API reference must also state the backend delegation invariants: config-driven ABACUS workflows retain external-abacuslite-first resolution; embedded calculator APIs do not reconfigure a supplied calculator; no abacuslite I/O API is re-exported; and ATST remains a workflow wrapper rather than a replacement for ABACUS, abacuslite, or job scheduling.

The optional ABACUS ASE-interface script remains a downstream consumer after an ATST release. Its header may link to the ATST API reference and CLI guide, but complete installation, MPI, YAML, and API behavior documentation remains owned by ATST-Tools.

Error handling and MPI

The API raises typed exceptions derived from ATSTAPIError: configuration validation, unsupported dependency, MPI configuration, and workflow execution failures. Each preserves the original exception as its cause and carries the workflow name and diagnostic context. The CLI translates these errors into its existing messages and exit behavior; it does not expose new API exception text as a changed user contract.

The API never starts an MPI launcher or scheduler. Under an existing launcher, all ranks call the same API with equivalent configuration. With world=None, the service uses the current ASE/MPI world. A supplied communicator is an embedding and test hook, not a scheduler API.

  • NEB validates that the world size equals the number of interior images.
  • AutoNEB validates that the world size equals n_simul.
  • When parallelism is requested with one rank, behavior remains the current serial fallback and warning.
  • Only root rank returns in-memory final_images and ts_atoms; all ranks receive identical status, metadata, and durable artifact paths.
  • Rank-local failures must be synchronized into an API execution failure before subsequent collective operations, preventing ranks from waiting indefinitely at barriers or reductions.

Testing and examples

  • Import-contract tests verify the supported names from atst_tools.api and prevent accidental exports.
  • Schema parity tests run every example YAML through CLI-equivalent and API normalization, including ABACUS and DP variants.
  • Dispatch parity tests cover dry-run and mocked execution for every existing workflow type.
  • CLI regression tests retain their existing calls and assert identical text, exit behavior, produced paths, and normalized YAML.
  • CCQN API tests verify calculator injection, input-copy semantics, explicit and automatic reactive modes, result metadata, and artifacts.
  • Backend delegation tests verify external-abacuslite-first fallback behavior for configuration-driven runs, assert that calculator injection never invokes CalculatorFactory, and verify backend_source metadata for both paths.
  • NEB/AutoNEB tests cover serial behavior, communicator-size mismatch, root/non-root result shape, endpoint synchronization, and optional real mpi4py launcher execution.
  • Package tests build the wheel, import the public API from the wheel, and run the API documentation example with a lightweight calculator fixture.
  • Documentation tests execute public import snippets and quick-start code, verify API-reference links from README and the documentation index, and reject examples that import non-public ATST modules.

ATST-Tools adds its own maintained Python API example and reference. After an ATST release containing the API, the ABACUS PR may add one concise optional script that creates an abacuslite calculator and invokes the CCQN API. That upstream script documents its purpose, optional installation, and links to ATST documentation, but does not introduce a build or runtime dependency for ABACUS itself.

Rollout and compatibility

  1. Extract internal application services behind the existing CLI without changing CLI parsing or output behavior.
  2. Add the public configuration API and make atst config validate and atst run call the same services.
  3. Add WorkflowResult, typed API errors, and configuration-driven API parity tests.
  4. Add the embedded CCQN API with calculator injection and its example/tests.
  5. Publish the API reference, README/index/user-guide navigation, CLI/API boundary, compatibility policy, and tested CCQN example in ATST-Tools; release the package.
  6. Only then update the optional ABACUS ASE-interface example, using the released import path and version requirement.

Rollback is additive: callers can continue using every existing CLI command and YAML file. If the new API has a defect before it is adopted externally, its public documentation can be marked provisional or the release withheld without reverting the current CLI surface.

Risks and resolved questions

Risk: CLI/API behavior drift. Mitigation: one service layer, example-wide parity tests, and a hard compatibility gate before release.

Risk: MPI deadlock or root-only result confusion. Mitigation: explicit communicator contract, collective failure handling, root ownership fields, and MPI-focused regression tests.

Risk: public API scope expands to every internal helper. Mitigation: only configuration-driven execution and CCQN embedding are stable in the first release; all other imports remain internal.

Risk: API exists but users cannot discover or safely adopt it. Mitigation: documentation and an executable example are release gates, with maintained links from existing user entry points and tested snippets from the installed wheel.

Risk: ABACUS example creates an unwanted dependency. Mitigation: keep it optional, one-file, and post-release; ABACUS does not import or package ATST outside that example.

Resolved decisions: public namespace is atst_tools.api; all executions use WorkflowResult; NEB path results and MPI semantics are first-class; first embedded workflow is CCQN; existing CLI and module contracts are unchanged.
